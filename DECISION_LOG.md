# DECISION_LOG.md — Sibyl Hub

Architectural decisions made during design and build, with rationale. The goal of this file is to make sure that future Alan, future Claude sessions, and any reviewer can reconstruct *why* something was built the way it was, not just *what* was built.

Each entry has a date, a topic, the decision, and a one-or-two-sentence rationale. Reversing a decision means appending a new entry, not editing the old one — the trail of reasoning matters.

---

## Pre-build (design conversation)

- **Domain & framing.** Sibyl Hub is "GitHub for agent workflows," with prediction-market trading as the v0 domain. The "compose your own analyst" framing is the v0 user story; "competing agents" is one level up, a feature users can build with the tooling rather than the premise of it. *Rationale:* tooling products generate their own content as users surprise the platform; tournament/leaderboard products are content products gated on a curator. Aligns with the platform-owner-blogs-the-findings model.
- **Polymarket-only for v0.** Kalshi and other prediction markets deferred. *Rationale:* deep familiarity with Polymarket from Apacen; data model designed so a Kalshi adapter is a clean later addition.
- **Paper trading only for v0.** Real-money trading is v1. *Rationale:* eliminates wallet management, custody, and runaway-bot blast radius for the v0 demo; architecture is cleanly capable of swapping in a real executor later.
- **Hetzner over EC2 for production VPS.** *Rationale:* flat ~$5/month with no data-transfer surprises (the Apacen lesson); identical Debian/systemd/Nginx/Cloudflare/Certbot pattern. Stays out of the AWS billing event horizon.
- **No third-party services beyond Anthropic and Polymarket.** No SMTP, no analytics, no Sentry/Datadog. *Rationale:* friction-of-suspicion against compounding vendor lock-in; Apacen's surprise was an external dependency. v0 has the budget to do everything in-house.
- **Single monorepo, not split repos.** *Rationale:* the original Apacen frontend/backend split made cross-cutting bugs annoying to fix; co-location with Claude Code is much smoother in a single repo.

## Renaming

- **2026-05-01: project renamed from "Oracle Garden" to "Sibyl Hub"** (mid-Phase 1, before any code beyond migrations had shipped). *Rationale:* "Oracle" collides with Oracle Corporation; we don't want any branding association with that company. "Sibyl Hub" preserves the spirit of the original framing — many divinatory voices in one place — without the corporate baggage. The mythological frame is now: a single agent is a *sibyl* (a prophetess delivering oracular advice), and the platform is the *hub* where many sibyls live and operate. Workflows remain "workflows" (we considered "consultations" and decided against — workflows can take independent action, not just report back). The terminal `trading_decision.v1` may be referred to as a *responsum* in code-comment flavor, but UI text uses "response" or "trading decision" plainly. *How to apply:* never reintroduce "Oracle Garden" or "oracle-garden" in committed code/docs; use "sibyl" lowercase when reaching for the divinatory metaphor in inline comments or naming.

## Architecture

- **Go (Chi, pgx/v5, golang-migrate, robfig/cron) + PostgreSQL + Next.js 14 App Router.** *Rationale:* matches the proven Maestro stack; Go concurrency suits the graph executor and SSE broadcaster.
- **No Goose runtime — direct Anthropic Messages API calls.** *Rationale:* Maestro's Goose integration cost significant debugging on output parsing and prompt-shape coherence. Direct API calls give cleaner structured-output validation, native tool-use round-tripping, and provider-agnosticism for v2+ when users want OpenAI/Gemini/local models.
- **JSON Schema (Draft 2020-12) as type-system source of truth.** AJV in the frontend (browser, runtime), `santhosh-tekuri/jsonschema/v6` on the backend. No build-time Zod generation. *Rationale:* type registry must support runtime registration of user-defined types in v2+; AJV is JS-only so backend uses the equivalent Go library.
- **`additionalProperties: false` on all v0 core schemas.** *Rationale:* a misnamed field in agent output is exactly the kind of bug that hides when extras are tolerated; cost of strictness is low.
- **`input_types` is "accepts," not "requires."** A node may declare types it can handle; the workflow's edges + `join_strategy` determine which subset is actually present at runtime. *Rationale:* cleanest way to support optional loop-back inputs (e.g., Thesis Builder accepting `risk_assessment.v1` only on iteration ≥ 2) without SCC analysis or special-case loop edges.
- **`approved` / `rejected` edge conditions restricted to `risk_assessment.v1` sources.** Validated at workflow save. *Rationale:* prevents the silent-failure mode of wiring `approved` to an output without an `approved` field.
- **Substring matching for custom edge conditions in v0; structured `{field, op, value}` conditions deferred to v1.** *Rationale:* substring is brittle but adequate for v0; structured conditions deserve their own design pass.
- **Replay buffer in the SSE broadcaster, not startup delay.** *Rationale:* eliminates the SSE-attachment race cleanly; engine fires its first agent step immediately. Maestro's 2-second delay was a hack worth retiring.
- **Drop `version` INT columns from `agent_templates` and `workflows`.** *Rationale:* versioning happens via fork (`forked_from`); the bare `version` field had no associated logic and was noise.

## Auth

- **NextAuth credentials provider with JWT session strategy.** No `sessions` table. JWTs encoded HS256 via overridden `jwt.encode` / `jwt.decode` (using `jsonwebtoken`, not NextAuth's default JWE). *Rationale:* Go can verify HS256 with `golang-jwt/jwt/v5` standard library; JWE would require `go-jose` and more ceremony. Trade-off: no server-side revocation in v0; cookie clear is sufficient for paper-trading platform.
- **`NEXTAUTH_SECRET` is a UTF-8 byte string used directly as the HMAC key on both sides — no base64 decoding.** *Rationale:* matches `jsonwebtoken`'s default and Go's `[]byte(env)` pattern; mismatched encoding produces silent "invalid signature" errors. Documented in `.env.example`.
- **Single JWT issuance path: NextAuth only.** Go signup creates the user and returns 200; frontend then calls `signIn()` to get the JWT. *Rationale:* avoids drift between two issuers' claim shapes.
- **Go auth endpoints under `/api/users/*`, not `/api/auth/*`.** NextAuth's catch-all owns the latter. *Rationale:* avoids collision; `/api/users` for signup, `/api/users/verify-credentials` for the internal check NextAuth's `authorize` calls server-to-server.

## Engine & cost protection

- **5 system agents seeded:** Market Watcher, News Scout, Thesis Builder, Risk Assessor, Paper Executor. **Demo custom-agent is Devil's Advocate** (built by Alan during the demo, not seeded). *Rationale:* replaces the originally-considered Contrarian Filter, which would have required the Polymarket Data API (not in v0).
- **Happy-path workflow** as the seeded demo: Watcher + Scout → Thesis → Risk → (approved) Executor / (rejected loop) Thesis. *Rationale:* exercises every engine feature (branching, loop, fan-in) with a meaningful end-to-end story.
- **Cost protection caps:** 50 runs/user/day, $5/user/day, 5-min minimum schedule interval, kill switch, 90-sec per-step timeout, 10-min per-run timeout, 50-step per-run cap, 6-round per-step tool cap, 1 validation retry. *Rationale:* the Apacen surprise was data transfer; Sibyl Hub's analogous risk is LLM spend. Bound it from every direction.
- **Atomic cost and run-count accounting via `INSERT … ON CONFLICT … RETURNING`.** *Rationale:* prevents two concurrent runs by the same user from both passing a pre-check and double-spending the cap.
- **Per-resource access rules:** strictly-owned (`runs`, `paper_trades`, `agent_steps`, `user_usage_daily`) require `user_id` match; shareable (`agent_templates`, `workflows`) allow `is_system` and `visibility IN ('public','unlisted')` reads. *Rationale:* system templates need to be readable by all users; runs are sensitive.
- **`paper_trades.entry_price` is the executable price for `'open'` rows and a midpoint snapshot for `'abstained'` rows.** Column not renamed. *Rationale:* analytics queries should filter by status; the term `entry_price` is natural for v1 real-money trading.

## Deferred to v1+ (with reasons noted in CLAUDE.md "Out of Scope")

Real-money trading; backtesting against historical Polymarket data; price/event triggers; public/unlisted UI; forking UI; user-contributed tools; email verification + password reset + OAuth; JWT revocation; Kalshi integration; WebSocket-driven real-time market subscriptions; production deployment; comparative analytics & leaderboards; structured edge conditions; build-time Zod generation; prompt-evaluation regression harness.

- **Dynamic per-model pricing in `pricing.go`.** v0 hardcodes per-MTok input/output rates for the three models chosen in Phase 0; v1+ should fetch pricing from Anthropic's `/v1/models` endpoint with a 24h cache and last-known fallback. *Why defer:* pricing has been stable enough that hardcoding works for v0 demos, and the runtime already measures actual `usage.input_tokens` / `usage.output_tokens` from API responses — so tokenizer differences (notably Opus 4.7's new tokenizer producing up to ~35% more tokens per request than Opus 4.6) are already reflected in cost calculations. Only the per-token rate constants are stale-able, and the failure mode of stale constants is at most cents per run.

### Phase 1 — Database, migrations, type registry seed (2026-05-01)

- **No deviations from CLAUDE.md / DATABASE_SCHEMA.md / TYPES.md.** Every table from DATABASE_SCHEMA.md is in `000001_init.up.sql` verbatim with the same column types, defaults, FK behavior, and indexes. The 6 core type schemas in `000002_seed_core_types.up.sql` match TYPES.md byte-for-byte (dollar-quoted JSON literals — no escaping changes the schema content). `additionalProperties: false` is present on every core schema, including the nested headlines item object in `news_digest.v1` (verified via `psql` after migration).
- **`pgcrypto` extension enabled in 000001_init.up.sql.** Postgres 16 ships `gen_random_uuid()` in core, but `CREATE EXTENSION IF NOT EXISTS pgcrypto;` is harmless and works on older managed hosts. *Why:* belt-and-suspenders for production hosting we haven't picked yet. *How to apply:* if a future migration uses `pgcrypto`-only functions (e.g., `digest()`), no extra setup required.
- **`paper_trades` FKs intentionally have no `ON DELETE CASCADE`** (matches DATABASE_SCHEMA.md verbatim). *Why:* paper trades are audit records; they should not silently vanish if a parent run row is deleted. v0 never deletes runs anyway, so RESTRICT is harmless and forward-compatible. *How to apply:* if v1+ ever needs to delete runs, write an explicit cleanup that handles the paper_trades rows first; do not retrofit CASCADE on this FK.
- **`migrate-down` defaults to `down -all`.** Reverts every applied migration without an interactive prompt. *Why:* STEPS.md Phase 1 verification expects `make migrate-down` to "cleanly revert"; the default `migrate down` (no args) prompts for confirmation, which breaks scripted use. *How to apply:* if a single-step rollback is needed during dev, run `migrate -path api/migrations -database "$DATABASE_URL" down 1` directly. The Makefile help mentions this.
- **Local dev convention: stop `maestro-postgres-1` while running Sibyl Hub's postgres on host port 5432.** Both projects bind 5432 by default; running them concurrently isn't supported on this machine. *Why:* Sibyl Hub is the Maestro successor (Alan's confirmation, 2026-05-01) — they aren't expected to run side-by-side, and keeping the standard `localhost:5432` convention in `.env.example` is worth more than the ability to dual-boot. *How to apply:* `docker stop maestro-postgres-1` before `docker compose up -d postgres` here; `docker start maestro-postgres-1` later if needed. Not a project-config decision — only flagged so future-me doesn't waste time re-deciding the port mapping.
- **Migration verification (recorded for reproducibility):** `docker compose up -d postgres` healthy in <2s; `make migrate-up` applies 3 migrations cleanly against the `sibyl_hub` database; `psql \dt` shows 12 expected tables + `schema_migrations` and **no `sessions` table**; `SELECT name, version FROM type_definitions ORDER BY name` returns the 6 core types; `SELECT key, value FROM system_config` returns `kill_switch | false`; `make migrate-down` reverts to a clean DB (only `schema_migrations` left); `make migrate-up` re-seeds back to the verified state. Initial verification ran against the pre-rename `oracle_garden` DB; the rename was applied to all migration files, .env.example (`postgres://sibyl:sibyl@localhost:5432/sibyl_hub`), docker-compose.yml (container `sibyl-hub-postgres`, role `sibyl`), and the run was repeated against the renamed DB before commit.

### Phase 2 — Polymarket adapter (2026-05-02)

- **Module bootstrapped at `github.com/OldEphraim/sibyl-hub/api`** with `go 1.22` directive (matches CLAUDE.md's "Go 1.22+"). Dependencies: `github.com/hashicorp/golang-lru/v2 v2.0.7`, `golang.org/x/time v0.5.0`. *Why x/time pinned to v0.5.0:* the latest v0.15.0 requires Go 1.25; pinning to v0.5.0 keeps the floor at 1.22 as documented. *How to apply:* don't `go get` the latest x/time without first revisiting the Go-version floor.
- **Stringified-number coercion via custom `UnmarshalJSON` on `*Market`, not `json.Number`.** Polymarket's Gamma API stringifies `liquidity` and `volume`, JSON-encodes-as-strings the `outcomes` / `outcomePrices` / `clobTokenIds` arrays (i.e., the value is a `"..."` whose contents are `[\"Yes\",\"No\"]`), and uses typed numbers for newer fields like `volume24hr`. `json.Number` doesn't help with the JSON-encoded-string-of-array case, and per-field tags would balloon the struct. Solution: a private `marketWire` mirror with three flex types (`flexFloat`, `flexStringArray`, `flexFloatArray`) that each accept all observed wire forms, plus a thin `UnmarshalJSON` on `*Market` that copies into the public struct's native types (`float64`, `[]string`, `[]float64`, `time.Time`). *How to apply:* if a new Gamma field exhibits the same drift pattern, reuse the flex types. Don't expose them in public APIs — keep them internal so callers always see native Go types.
- **Rate-limit library: `golang.org/x/time/rate` (token bucket).** Default 50 req/min, burst 5; configurable via `ClientOptions.RatePerMin` / `RateBurst` or a fully-pre-configured `*rate.Limiter` for tests. *Why:* CLAUDE.md prescribes "token-bucket limiter" and 50 req/min; x/time is the standard-library-adjacent choice — well-tested, no third-party dependencies, and tests can wire `rate.NewLimiter(rate.Inf, 1)` to disable rate limiting deterministically.
- **Single in-memory LRU keyed by full request URL+query, with per-entry TTL.** TTL profiles match CLAUDE.md: 5 min metadata, 30 sec prices, 60 sec orderbooks. Cache miss path emits a DEBUG log with upstream latency; cache hit path emits an INFO log (`polymarket cache hit`) so `pmctl` shows it without needing `-v`. *How to apply:* if a method needs a different TTL, add a constant to `cache.go` rather than passing magic durations from `methods.go`.
- **429 backoff: exponential with up to 25% jitter, honors Retry-After when present.** Capped at 4 retries (worst-case ~4s before giving up). Other 4xx/5xx responses surface as `*errHTTPStatus` without retry — use `polymarket.IsNotFound(err)` to branch on 404.
- **Polymarket API drift documented (CLAUDE.md said X, live behavior is Y):**
  1. **`GET /markets/{slug-or-id}` rejects slugs with HTTP 422 ("id is invalid").** Only numeric IDs work in the path form. The adapter resolves slugs via `GET /markets?slug=<exact>&limit=1` (the list-with-filter form), which works correctly. Documented in a code comment on `GetMarket` so the next reader doesn't burn time on a 422.
  2. **`GET /search` requires authentication** ("invalid token/cookies" without auth) despite CLAUDE.md describing it as public. **However, Polymarket also exposes `GET /public-search`, which does NOT require authentication** — see the Phase 2.5 entry below for the corrected SearchMarkets implementation. The adapter does not call `/search` directly.
  3. **`/markets` silently accepts and ignores unknown query params** (e.g., `slug_contains`, `q`, `slug_starts_with`). Returns the default-sorted list as if the param weren't there, with no error. There's no way to distinguish "filter applied, no results" from "filter unknown, default list". The exact-slug path (`?slug=`) is the only filter we trust. *How to apply:* see Phase 2.5 — the adapter now enforces a hand-curated allowlist of Gamma /markets keys and WARN+drops anything else, so callers get a visible signal instead of silently-misleading data.
- **Live verification on 3 markets** (high-liquidity, different domains, on 2026-05-02): `will-bitcoin-hit-150k-by-june-30-2026` (crypto, YES midpoint 0.0175), `nba-phi-bos-2026-05-02` (sports, 76ers midpoint 0.3250), `ufc-car19-jac9-2026-05-02` (sports, Carlos Prates midpoint 0.9995). All three: clobTokenIds extracted correctly, outcomes/outcomePrices coerced to `[]string` / `[]float64`, midpoints in [0,1], cache-hit log fires on second call.
- **`pmctl` cache-hit demonstration is intra-process.** STEPS.md says "re-running pmctl logs a cache hit" — but the LRU is in-memory, so that's only true within a single process. `pmctl` therefore makes each call twice in the same run; the second call is the one that emits the cache-hit log. Documented in a comment on `pmctl`'s `run` function. (Persistent caching is not warranted for v0.)
- **Tests at end of Phase 2: 9** (stringified-number coercion incl. nested JSON-encoded arrays, cache hit, cache TTL expiry, 429 retry, token_id-vs-conditionId routing, orderbook coercion, price-history typed-number decode, non-retriable 4xx, exact-slug regression flag). The exact-slug regression flag was replaced in Phase 2.5; see below for the current test inventory (14).

### Phase 2.5 — `/public-search`, global User-Agent, `/markets` allowlist (2026-05-02)

Mid-Phase-2 fix: a sibling project's Python helper (in `polymarket-go-connection`) revealed that Polymarket exposes **two** search endpoints — `/search` (auth required) and `/public-search` (no auth) — and the latter is what we actually want. Phase 2 had documented `/search` as auth-gated and shipped `SearchMarkets` as exact-slug-lookup-only; Phase 2.5 corrects this and tightens two other rough edges discovered alongside.

- **`SearchMarkets` now hits `GET /public-search`** (Gamma) — no auth, real free-text search. The endpoint requires a User-Agent header (some edge nodes 403 without one); see below. Phase 10 (Markets browser) can use this directly without a curated DB-backed catalog. Live verification on 2026-05-02 against `q=trump`, `q=bitcoin`, `q=election` returned 5/5/0 open markets respectively (zero for "election" is the legitimate result — no currently-active market with that title). *How to apply:* the distinction between `/search` and `/public-search` is non-obvious from public Polymarket documentation; the working pattern was found by inspecting a sibling project's HTTP traffic. If `/public-search` ever starts requiring auth, fall back to a local index seeded from periodic `/markets` snapshots before standing up an authenticated search adapter.
- **Two `/public-search` quirks worth recording so future-me doesn't relearn them:**
  - The `keep_closed_markets` param expects `0`/`1`, not `true`/`false` — passing booleans returns HTTP 422 "wrong value".
  - Even with `keep_closed_markets=0`, the filter is applied at the **event** level, not the market level. Events whose markets are all closed get dropped, but markets within a still-open event can themselves be closed. `SearchMarkets` therefore client-side filters `m.Closed == true` after walking the nested `events[].markets[]` tree.
  - The response's top-level shape is `{events: [...], pagination: {}}`, not `{markets: [...]}`. There is no top-level `markets` field.
- **Global User-Agent on every outbound request.** New `ClientOptions.UserAgent` field; default is `"sibyl-hub/0.1 (+https://github.com/OldEphraim/sibyl-hub)"`. *Why:* `/public-search` requires it on at least some edge nodes, and identifying ourselves is good citizenship — Polymarket can rate-limit by client identity rather than blanket-rate-limiting all unauthenticated traffic, which protects legitimate users in our position. *How to apply:* the UA is stored on the client and applied uniformly in `fetchWithRetry`; do not hardcode it in any other call site. If we ever publish a v1.0, bump the version string here.
- **Defensive allowlist on Gamma `/markets` query params.** `client.go` defines `allowedMarketsParams` and a `gammaMarketsURL(values url.Values) string` helper. Unknown keys are dropped with a WARN log (`polymarket: dropping unrecognized /markets query param — silently ignored by API`). The endpoint itself does not reject unknown keys (Phase 2 finding #3), so without this guardrail a future caller writing `?question_contains=...` would get the unfiltered default list back and assume the filter ran. *How to apply:* all `/markets`-targeted calls in this package (currently only `GetMarket`) MUST go through `c.gammaMarketsURL`. The allowlist is intentionally hand-curated; expand only after verifying the new key actually filters against a live endpoint. Other endpoints (`/public-search`, `/events`, CLOB) are not subject to this allowlist — they have their own param shapes.
- **Tests after Phase 2.5: 14** (added `TestPublicSearchExtractsMarkets`, `TestSearchMarketsEmptyResults`, `TestSearchMarketsRespectsLimit`, `TestUserAgentOnEveryRequest`, `TestUserAgentDefault`, `TestMarketsAllowlistDropsUnknown`; removed `TestSearchMarketsExactSlugLookup` since it documented the now-obsolete behavior). `go test ./internal/polymarket/...` passes in ~0.6s. `pmctl` regression check passed against `will-bitcoin-hit-150k-by-june-30-2026` post-changes.

### Phase 3 — Type registry, JSON Schema validator, repositories (2026-05-02)

- **Test strategy: transactional rollback against a shared test DB, not testcontainers.** Each test runs inside a tx that's rolled back at teardown; the package-wide `withTx(t, fn)` helper in `repos_test.go` is the entrypoint. *Why:* fast (no per-test container provisioning), deterministic (every test sees the same migration-seeded baseline), no Docker required for `go test`, and `pgx.Tx` satisfies the same `db.Querier` interface as `*pgxpool.Pool` — so the production code path and the test code path are bit-identical. *How to apply:* every new repo test must use `withTx`; never call `pool.Begin` directly. Tests requiring a user FK go through `mustCreateUser(t, ctx, tx)`, which inserts a user with `uniqEmail()` so duplicate-email tests don't collide across reruns.
- **Test connection: `TEST_DATABASE_URL` falls back to `DATABASE_URL`; both empty → `t.Skip`.** Local devs run against the same postgres they migrated; CI sets `TEST_DATABASE_URL` to an isolated DB. The test pool is process-wide (sync.Once), capped at 4 conns. *How to apply:* if a future CI gets a dedicated test DB, set `TEST_DATABASE_URL` only there — don't break the local fall-through.
- **Querier interface in `internal/db/db.go`** is the union of `pgx.Query/QueryRow/Exec`. Both `*pgxpool.Pool` and `pgx.Tx` satisfy it. Every repo function accepts `q db.Querier`. The single exception is `SubscriptionsRepo.SyncForWorkflow` (and `ListForWorkflow`), which accept `pgx.Tx` directly — its delete-then-insert contract is meaningless without a tx, and a typed parameter makes the intent unmissable. *How to apply:* future repos follow the Querier convention. Reach for `pgx.Tx` in a signature only when the operation is invalid outside a tx.
- **Type registry: load-once at startup, compile-once, atomic Reload.** `types.Load(ctx, q)` reads all `type_definitions` rows and compiles each schema with `santhosh-tekuri/jsonschema/v6`. Get/ValidateAgainst hit the cached compiled schemas via an RWMutex; no per-call re-parsing. `Reload` rebuilds the map atomically (in-flight readers see the prior compiled schema until they next take the read lock). *Why:* CLAUDE.md says "compile JSON Schema validators once, cache them"; ValidateAgainst is on the per-step hot path during a workflow run. *How to apply:* the registry is process-singleton — pass the same `*types.Registry` to every consumer (currently `AgentTemplatesRepo`; in Phase 4+ the agent runtime; Phase 8 the `/api/types` handler). v0 doesn't reload at runtime; if v1+ supports user-registered types via the API, call `Reload` after the INSERT commits.
- **AgentTemplatesRepo holds `*types.Registry` and validates input/output types at create AND update time.** Per the user spec — putting validation at the persistence boundary (rather than in handlers) makes "you can't reference an unregistered type" an invariant nobody can forget. `MustExist` reports ALL missing types in one shot, so a UI form that submits three bad type names learns about all three at once. *How to apply:* if Phase 8 introduces a service layer between handlers and repos, the registry stays on the repo — handlers shouldn't have to remember to call `MustExist`.
- **Repo error conventions:** missing rows → `db.ErrNotFound`; pgx's `ErrNoRows` is normalized via `db.MapNoRows`. Constraint violations (unique, FK) surface raw pgx errors — handlers branch with `db.IsUniqueViolation(err)` / `db.IsForeignKeyViolation(err)`. ### Phase 4 — Agent runtime (Anthropic API client) (2026-05-03)

- **SDK choice: hand-rolled HTTP client, not `anthropic-sdk-go`.** `runtime/client.go` is ~150 lines: POST `/v1/messages`, three headers, typed `MessagesRequest` / `MessagesResponse` / `Usage` / `AnthropicError`. *Why:* our v0 surface is tiny (text-in, text-out + Phase-5 tool_use) and the official SDK pulls in streaming, batches, vision, files, MCP, and citations — capabilities we don't use and don't want to keep up with. The hand-rolled client also makes test injection (`BaseURL` override pointing at `httptest.NewServer`) trivial; pinning a third-party SDK would force us to either intercept its `*http.Client` or run a fake at the network layer. Trade-off: when Anthropic adds a new feature we want, we write the structs ourselves rather than pulling a tagged release. Worth it at our scale.
- **Retry-on-validation-failure: single shot, augmented system prompt** (CLAUDE.md "validation retry cap" — 1 retry, 2 attempts total). The retry's system prompt = original system + a NOTE block carrying:
  1. The previous response, truncated to 4 KB (so a runaway response can't blow the next prompt past the model's context).
  2. Up to 5 of the validator's structured error messages, formatted as bullets.
  3. A short instruction telling the model not to repeat the previous mistake.
  *Why this shape:* re-sending the same prompt without context just produces the same failure; including the failed text shows the model what it actually said, including hallucinated fields it might re-emit. Including only the validator messages (without the failed text) would leave the model guessing where in its output the issue was. CLAUDE.md prescribes "system note" as the carrier, not a follow-up user-message turn — option-A vs option-B in design — option-A keeps the conversation single-shot and avoids doubling input tokens. *How to apply:* don't grow the truncation cap; if a model emits >4 KB of broken output, it has bigger problems than format-correction can fix. The accumulated PromptTokens/CompletionTokens reflect both attempts so the cost cap stays accurate.
- **Schema lives in the user message, not the system prompt** (matches CLAUDE.md "Per-step flow" verbatim). Originally drafted with the schema in the system prompt; corrected to keep the literal schema adjacent to the data being described. The system prompt only carries the agent's role + the structured-output instruction + (when multi-input) a description of the upstream-keyed input shape. *Why:* keeping the schema in the user message means Phase-5 tool-use rounds (which extend `Messages` with assistant + tool_result turns) don't have to re-state the schema in the system; it's already in the original user message and stays in the conversation history.

### Phase 5 — Tools registry + tool-dispatch loop (2026-05-05)

- **Carried-over Phase 4 fix-up: cost is now computed from `result.Model` (the response-echoed dated form), not `agent.Model`.** Fixes the alias→dated-snapshot mismatch where calling `claude-haiku-4-5` would silently 0-cost despite resolving server-side to the priced `claude-haiku-4-5-20251001`. The pricing table doesn't need to carry both alias and dated forms.
- **Tool-dispatch state machine.** `runtime.generateOnce` now loops up to `MaxToolRounds = 6` (CLAUDE.md cost-protection cap). Each iteration: call API → if `stop_reason != "tool_use"` OR no `tool_use` blocks present, return the final text → otherwise dispatch every local `tool_use` block, append the assistant's full content + the user-side `tool_result` blocks to the message list, loop.   *How to apply:* don't bypass the cap. If a future agent legitimately needs more rounds, it's a sign the agent should be split into multiple workflow steps.
- **6-round cap interaction with `stop_reason`.** Three exit states from the loop:
  1. `stop_reason in {"end_turn","stop_sequence","max_tokens"}` — final response; extract text and return.
  2. `stop_reason == "tool_use"` AND no `tool_use` blocks — anomalous; log a WARN and treat as final (don't loop forever waiting for blocks that aren't there).
  3. `stop_reason == "tool_use"` on round 6 — cap hit; return error `"runtime: tool-use cap reached (6 rounds) without final text"`. The error is NOT a validation failure, so `Invoke`'s validation+retry wrapper does not retry. Tokens accumulated through the failed rounds are still surfaced on `InvokeResult` for cost accounting (already-paid-for tokens).
  *Test coverage:* `TestToolUseCapAtSixRounds` asserts exactly `MaxToolRounds` API calls and `MaxToolRounds-1` tool dispatches when the model loops indefinitely.

The model can then react — abandon the call, try a different input, or abstain. *Why:* a single flaky tool call shouldn't crash the runtime; the per-step retry budget already guards against the catastrophic case. *How to apply:* the only conditions that propagate as runtime errors out of `dispatchToolUseBlocks` (none, in v0) would be programmer bugs — every other failure is delivered to the model as data.

Anthropic-facing definitions and tool_use response routing use underscores.

### Phase 6 — Workflow execution engine (2026-05-07)

- **Engine state machine.** Per-node state on the in-memory `Node`: `Iteration`, `LastFiredSeq`, `LatestUpstreamOutputs[upstream_node_key]`, `LastOutput`, `Pending`. A run-wide `seq` counter increments on every propagation event (entry-seeding, edge-following) so each upstream output carries a per-arrival sequence number. The "is there something new since last firing?" gate (`max(LatestUpstreamOutputs.seq) > LastFiredSeq`) is what unifies `all` and `first` join strategies — they only differ on the iteration-1 readiness check. *Why:* the spec says `all` requires "every upstream has produced ≥1 output total" for iter 1 and "a new output arrived at any upstream" for subsequent iters. Sequence numbers express both as a single condition with one tweak per strategy. Same pattern handles loop iterations naturally.
- **Loop-edge detection at graph build, via DFS back-edge analysis.** Without this, `all` join's iter-1 readiness counts loop-edge upstreams (e.g., Risk → Thesis), and Thesis can never fire on iter 1 because Risk hasn't produced yet — the engine deadlocks at run start. Fix: `markLoopEdges` does a DFS from entry nodes with white/gray/black coloring; any edge pointing to a gray node is flagged `IsLoop`. `Node.NonLoopUpstreamKeys` then drives the iter-1 readiness count for `all` join. Per CLAUDE.md "Loop semantics" worked example, this is exactly what the happy path expects: "Iteration 1 of Thesis Builder: triggered when Watcher and Scout have both produced output." (Loop-edge upstream Risk doesn't count for iter 1.) *How to apply:* tests that use cycles in their workflow definitions need `loop_iteration_limit > 1` on every node touched by a loop, not just the loop target — Risk in the happy path is fired once per Thesis iteration, so its limit must match Thesis's. Caught during `runctl` verification when Risk's default limit-of-1 fired the cap before Thesis hit its own limit.
- **Edge-evaluation rule (CLAUDE.md worked example, encoded in `edgeFiringPlan`).** Edges arrive in `(priority ASC, id ASC)` order. Walk: every `always` edge is added to the firing plan; among non-`always` edges, the first one whose condition matches is added; remaining non-`always` edges are suppressed. The matched-conditional flag latches once and is checked before each conditional eval. Test `TestMixedFanOutAndBranching` exercises the worked example shape (always priority 0, approved priority 1, rejected priority 2) and asserts logger fires AND executor fires AND fallback doesn't.
- **Single ready-queue, FIFO semantics, with `Pending` flag preventing duplicate enqueueing.** A node is added once when its inputs first satisfy `isReady`; it's removed when popped to fire. The "post-fire re-evaluation" needed for `first` join's multi-firing is handled by the FIFO ordering itself — when an upstream propagates a new output to an already-pending downstream, the propagation sets `LatestUpstreamOutputs` and the queued firing picks up the new max-seq naturally. *Subtlety surfaced during testing:* with two simultaneously-seeded entry nodes and FIFO order, a downstream `first`-join node pops AFTER both upstreams have propagated, so it fires once with both inputs — not twice. The original `TestFanInFirst` assumed two firings; restructured to a fan-out + chain shape (a → c, a → b, b → c) so c gets popped between a's and b's propagations and fires twice as the spec describes. Documented in the test's preamble.
- **Step persistence via `db.Querier`, not `*pgxpool.Pool`.** Same Querier convention as Phase 3 — engine tests pass a transaction that's rolled back at teardown, production passes the pool. The Phase 7 cost-quota refactor will wrap step finalization in an explicit tx alongside the user_usage_daily increment; the engine's contract doesn't change.

### Phase 6.5 — Happy-path edge fix + SSE replay-buffer cap (2026-05-08)

- **CLAUDE.md happy-path workflow gains a 6th edge: `Thesis → Executor`, `condition = always`, `priority = 1`.** Without this edge, the Paper Executor's merged input lacks `thesis.v1` — meaning the Executor never sees `market_slug` or `direction` and ends up abstaining with "no market slug." Phase 6.5 makes the spec match the working bootstrap so Phase 14's seed JSON inherits the correct shape.
  *How to apply:* the Executor now has TWO incoming edges (Thesis always, Risk approved). Its `join_strategy = all` is what preserves gating semantics — the Risk approved-edge activates the executor path, while the Thesis edge ensures `thesis.v1` is in the merged input. With `all`, the Executor only fires when both upstreams have produced output. - **STEPS.md Phase 8 SSE replay buffer gains a 500-event cap with overflow signaling.** Without a cap, a runaway loop (e.g. a future bug that doesn't trigger the loop-iteration-limit) could OOM the broadcaster; with a per-run buffer hard cap of 500 events and a `BufferTruncated` event emitted on overflow, late-attaching clients learn they've missed history and the broadcaster's memory footprint stays bounded. *Sizing rationale:* 50-step run cap × ~10 events per step (StepStarted, StepProgress×N tool rounds, StepCompleted, EdgeFollowed×fan-out) ≈ 500 max for a worst-case run; normal runs stay well under. *How to apply:* when implementing `broadcaster.go` in Phase 8, the ring buffer's max size and the `BufferTruncated` event type are non-negotiable — both the late-subscriber replay path and the live-stream path must emit `BufferTruncated` exactly once per overflow.
